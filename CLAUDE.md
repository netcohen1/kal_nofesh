# CLAUDE.md - קל נופש (אתר דירות נופש)

מסמך זה מנחה את בניית האתר. הוא נגזר משני מסמכים קיימים: **מסמך האפיון** ו**תוכנית הפיתוח**, ומשלב את ההערות שסוכמו מול הלקוח. כל החלטה בבנייה צריכה להיות עקבית עם המסמכים האלה ועם הקובץ הזה. במקרה של סתירה, הקובץ הזה גובר.

המטרה: לבנות אתר מלא לפרסום דירות נופש, מוכן לפריסה ישירה ל-Netlify, עם חלק ציבורי למתעניינים וחלק ניהולי למפרסם.

---

## 1. סקירת הפרויקט

- **שם המותג:** קל נופש.
- **מהות:** אתר שמרכז דירות נופש להשכרה, עם חיפוש וסינון לציבור, ואזור מפרסם לניהול הנכסים וההגדרות.
- **שני חלקים:**
  1. **חלק ציבורי (מתעניינים):** דף בית עם מנוע חיפוש, דף תוצאות, דף נכס, ויצירת קשר.
  2. **חלק ניהולי (מפרסם):** התחברות, ניהול נכסים (כולל תמונות), וניהול רשימות החיפוש והגדרות.
- **עיקרון מנחה מהאפיון:** רשימות הערכים שמזינות את החיפוש (סוג נכס, עיר, תקופה) ומגבלות המחיר ומספרי הטלפון נשלטות אך ורק בחלק הניהולי. המפרסם הוא שמגדיר אותן.

---

## 2. סדר עדיפויות מחייב

שלושה עקרונות מובילים כל החלטת תכנון וקוד. כשיש התלבטות, מכריעים לפיהם בסדר הזה:

1. **אמינות.** הנתונים נשמרים נכון ולא הולכים לאיבוד, הקלט נבדק בצד הלקוח ובצד השרת, יש טיפול בשגיאות ובמצבי קצה, ואין סודות בקוד הלקוח או בריפו.
2. **מודולריות.** קוד מחולק לפי תחומים (features), רכיבים קטנים ועצמאיים, שכבת גישה לנתונים מופרדת מה-UI, וטיפוסים ברורים. שינוי בתחום אחד לא שובר אחרים.
3. **מהירות.** דפים ציבוריים נטענים מהר, תמונות ממוטבות, מינימום JavaScript בצד הלקוח, ושאילתות יעילות.

---

## 3. התבנית העיצובית (קריטי)

- קיים **קובץ HTML של התבנית העיצובית** שיימסר לבונה האתר. הוא אינו מצורף לקובץ הזה, אבל **חובה לוודא שהוא קיים ולעבוד לפיו**.
- **להתייחס אך ורק לשפת העיצוב שבקובץ:** צבעים, טיפוגרפיה, מרווחים, פינות, צללים, רכיבים (כפתורים, כרטיסים, שדות), והפריסה הכללית.
- **אסור** להשתמש במלל, בתוכן, בנתונים, בשמות, במספרים או בכל מידע אחר שמופיע בקובץ ה-HTML. כל הטקסטים והתוכן באתר נלקחים מהקובץ הזה ומהאפיון בלבד.
- יש לחלץ מתוך ה-HTML **design tokens** (צבעים, גופנים, גדלים, מרווחים) ולהטמיע אותם כמשתני עיצוב מרכזיים (CSS variables / theme), כך שכל הרכיבים נשענים עליהם.
- אם משהו בתבנית סותר את האפיון מבחינה פונקציונלית, הפונקציונליות לפי האפיון, והעיצוב מתאים את עצמו.

## 4. ארכיטקטורה ומחסנית טכנולוגית

הכול רץ **בתוך הפלטפורמה של Netlify**, בלי שירותי צד שלישי חיצוניים. זו המשמעות המעשית של "הכול מקומי" בהערה 9: הקוד, הנתונים, התמונות וההתחברות חיים בתוך הפרויקט והפלטפורמה, לא בשירות חיצוני נפרד. הסודות בלבד נשמרים במשתני סביבה של Netlify (זו החריגה הביטחונית, כי אסור לשמור אותם בקוד או בריפו).

| שכבה | בחירה | תפקיד |
|------|-------|-------|
| אירוח ופריסה | Netlify (direct deploy) | אחסון האתר, build ופריסה |
| Frontend | מומלץ Astro (חלופה: Next.js) | דפים ציבוריים מהירים, RTL, islands למינימום JS |
| Backend | Netlify Functions (serverless) | API לקריאה ציבורית ולכתיבה מאובטחת מהאזור הניהולי |
| התחברות מפרסם | Netlify Identity | התחברות מאובטחת לאזור הניהולי (חלופה: תוסף Auth0) |
| נתונים מובנים | Netlify Database (Postgres) | דירות, ערים, סוגי נכס, תקופות, הגדרות |
| תמונות וקבצים | Netlify Blobs (+ Image CDN) | אחסון תמונות הנכסים ואופטימיזציה |
| סודות | Netlify environment variables | מחרוזת חיבור ל-DB, מפתחות. לעולם לא בקוד הלקוח ולא בריפו |

**הערות בחירה:**
- בחרנו Astro לחלק הציבורי בגלל המהירות (יצירה סטטית של דפים, שליחת מעט מאוד JavaScript). אפשר Next.js כחלופה אם נוח יותר לבונה, כל עוד נשמרים RTL, ביצועים, ושימוש בשירותי Netlify.
- **נתונים מול תמונות:** Netlify Blobs מיועד לקבצים (תמונות), לא לרשומות הניתנות לשאילתה. לכן הנתונים המובנים יושבים ב-Netlify Database (Postgres) שמתאים למודל הרלציוני ולסינון, והתמונות ב-Blobs. אם בעתיד היקף הנתונים יישאר קטן מאוד, אפשר לשקול שמירת JSON ב-Blobs, אבל ברירת המחדל היא Postgres.

**זרימה כללית:**
- ציבורי: הדפדפן טוען דפים ציבוריים (סטטיים/SSR) ⟶ קורא נתוני נכסים דרך Function או בזמן build ⟶ הסינון רץ בצד הלקוח על אינדקס מוכן או דרך שאילתה יעילה.
- ניהולי: המפרסם מתחבר (Netlify Identity) ⟶ פעולות כתיבה עוברות דרך Functions מאומתות ⟶ נתונים ל-Postgres, תמונות ל-Blobs.

## 5. מבנה הפרויקט (מודולרי, לפי תחומים)

המבנה לדוגמה (יותאם ל-framework שייבחר), אבל העיקרון קבוע: **חלוקה לפי תחום, הפרדת שכבת הנתונים מה-UI, ורכיבים קטנים לשימוש חוזר.**

```
/
├─ public/                      # נכסים סטטיים (לוגו, favicon, אייקונים)
├─ src/
│  ├─ styles/
│  │  └─ tokens.css             # design tokens שחולצו מתבנית ה-HTML
│  ├─ content/
│  │  └─ copy.he.ts             # כל הטקסטים בעברית במקום אחד (ראו פרק 9)
│  ├─ lib/
│  │  ├─ data/                  # שכבת גישה לנתונים (DB) - היחידה שמדברת עם Postgres
│  │  ├─ media/                 # עבודה מול Blobs (העלאה, קישור תמונה ממוטבת)
│  │  ├─ whatsapp.ts            # בניית קישורי ווצאפ (ראו פרק 7)
│  │  └─ validation/            # סכמות ולידציה משותפות (לקוח + שרת)
│  ├─ components/               # רכיבי UI קטנים לשימוש חוזר (כפתור, כרטיס, שדה)
│  ├─ features/
│  │  ├─ search/                # טופס חיפוש + לוגיקת סינון
│  │  ├─ listings/              # כרטיס נכס, רשת תוצאות, דף נכס
│  │  ├─ contact/               # זרימת ווצאפ, מודאל שם
│  │  └─ admin/                 # מסכי ניהול (נכסים, רשימות, הגדרות)
│  └─ pages/                    # נתיבים (דף בית, תוצאות, נכס, admin)
├─ netlify/
│  └─ functions/               # API: קריאה ציבורית + כתיבה מאומתת
├─ netlify.toml                 # build, functions, redirects
└─ CLAUDE.md
```

כללי מודולריות:
- רכיב UI לא ניגש ישירות ל-DB. הוא מקבל נתונים דרך props או דרך שכבת `lib/data`.
- כל תחום ב-`features/` עצמאי ככל האפשר וחושף ממשק ברור.
- סכמות הוולידציה משותפות בין הלקוח לשרת (מקור אמת אחד).

## 6. מודל הנתונים

נגזר מפרק 7 באפיון. שמות השדות באנגלית (לקוד), עם הסבר בעברית.

**Property (נכס / דירת נופש)**

| שדה | סוג | חובה | הערות |
|------|------|------|-------|
| id | מזהה | כן | נוצר אוטומטית |
| name | טקסט | כן | שם הדירה, משמש לחיפוש ולכותרת הכרטיס |
| type_id | קשר | כן | מתוך טבלת PropertyType |
| city_id | קשר | כן | מתוך טבלת City |
| neighborhood | טקסט | לא | שכונה / אזור בתוך העיר |
| period_ids | קשר רבים | כן | אחת או יותר מטבלת Period |
| price_per_night | מספר | כן | בתוך גבולות המחיר שבהגדרות |
| rooms | מספר | לא | מספר חדרים |
| beds | מספר | לא | מספר מיטות |
| description | טקסט ארוך | לא | תיאור הדירה |
| images | רשימת קבצים | כן | מאוחסנות ב-Blobs, תמונה ראשית + גלריה |
| phone | טקסט | כן | טלפון המפרסם לדירה זו |
| whatsapp | טקסט | כן | מספר ווצאפ לדירה זו, בפורמט בינלאומי (לדוגמה 9725XXXXXXXX) |
| status | בחירה | כן | `published` או `draft` |

**ישויות הגדרה (רשימות החיפוש):**

| ישות | שדות | הערות |
|------|------|-------|
| PropertyType | id, name | מזין את סינון "סוג נכס" ואת טופס הנכס |
| City | id, name, region (אופציונלי) | מזין את סינון "עיר", ניתן לקבץ לפי אזור |
| Period | id, name | מזין את סינון "תקופה" (לדוגמה: כל השנה, חגים, שבתות, בין הזמנים) |

**Settings (הגדרות גלובליות, רשומה אחת):**

| שדה | סוג | הערות |
|------|------|-------|
| currency | טקסט | מטבע התצוגה, לדוגמה ₪ |
| price_min | מספר | גבול תחתון לבורר המחיר |
| price_max | מספר | גבול עליון לבורר המחיר |
| publish_whatsapp | טקסט | מספר הווצאפ ל"פרסום דירה" בדף הבית, בפורמט בינלאומי |

**קשרים:** נכס משויך לסוג נכס אחד, לעיר אחת, ולתקופה אחת או יותר. עיר משויכת לכל היותר לאזור אחד.

**הערה על תקופה:** לפי האפיון, תקופה היא קטגוריה מוגדרת מראש שמנוהלת בחלק הניהולי (לא טווח תאריכים). מודל הנתונים תומך בהוספה עתידית של טווח תאריכים אם יוחלט.

## 7. פונקציונליות - החלק הציבורי

### 7.1 דף הבית
מבנה מלמעלה למטה: כותרת עליונה (לוגו, ניווט, גישה לחיפוש), אזור חיפוש בולט (Hero), אזור נכסים מוצגים (אחרונים או מומלצים), כותרת תחתית.

**טקסט הכותרות בדף הבית (מדויק, להשתמש כמו שהוא):**
- כותרת ראשית, בשתי שורות:
  - שורה 1: `מקום שמרגיש שלכם -`
  - שורה 2: `מהרגע שנכנסים.`
- כותרת משנית: `לסגור בקלות ובמהירות את דירת הנופש המושלמת שלכם.`

**קריאה לפעולה "פרסום דירה" (הערה 4):**
- בדף הבית, במקום **בולט אך לא חוסם**: לא חלון קופץ שמכסה תוכן ולא שכבה מטרידה. למשל פס/כרטיס ברור בזרימת הדף, או כפתור שאינו מסתיר תוכן.
- טקסט: `מעוניינים לפרסם את דירת הנופש שלכם? בא נדבר`
- בלחיצה: נפתחת שיחת ווצאפ אל המספר `publish_whatsapp` שמההגדרות, עם הודעה מוכנה (ראו 7.5).

### 7.2 מנוע החיפוש
לב האתר. כל רשימות הערכים מגיעות מהחלק הניהולי. כל השדות אופציונליים, והשילוב ביניהם הוא "וגם" (התוצאה עומדת בכל התנאים שנבחרו).
- **חיפוש לפי שם דירה:** טקסט חופשי, התאמה חלקית, ללא תלות באותיות גדולות/קטנות.
- **סינון לפי סוג נכס:** רשימה נפתחת מתוך PropertyType. ברירת מחדל "כל הסוגים".
- **סינון לפי עיר:** רשימה נפתחת מתוך City, אפשר מקובצת לפי אזור. ברירת מחדל "כל הערים".
- **סינון לפי תקופה:** רשימה נפתחת מתוך Period.
- **סינון לפי מחיר:** בורר טווח לפי מחיר ללילה, גבולותיו מ-`price_min`/`price_max` שבהגדרות.
- **התנהגות:** כפתור חיפוש מוביל לדף תוצאות, כפתור איפוס מנקה הכול, מוצג מספר תוצאות, ויש מצב "אין תוצאות" עם הצעה להרחיב את החיפוש.

### 7.3 דף תוצאות החיפוש
רשת/רשימה של כרטיסי נכס. כל כרטיס: תמונה ראשית, שם, סוג, עיר, תקופה, מחיר ללילה, חדרים/מיטות (אם קיימים), וכפתור "לפרטים נוספים". רצוי מיון (מחיר עולה/יורד, חדשים) וחלוקה לעמודים כשיש הרבה תוצאות. אפשר לעדכן את הסינון בלי לחזור לדף הבית.

### 7.4 דף נכס בודד
גלריית תמונות, שם, סוג, עיר ושכונה, מחיר ותקופות, תיאור ומאפיינים (חדרים, מיטות), ואזור יצירת קשר (ראו 7.5). מיקום על מפה אופציונלי.

### 7.5 יצירת קשר וזרימת ווצאפ
יצירת קשר לכל דירה היא **טלפון וּווצאפ בלבד** (אין טופס פנייה ואין דוא"ל).

**זרימת ווצאפ לדירה (הערה 2):**
1. הלקוח לוחץ על "שליחה בווצאפ".
2. נפתח שדה קצר שבו הלקוח **מזין שם**.
3. נפתחת שיחת ווצאפ אל מספר הווצאפ של הדירה (`whatsapp` מהנכס) עם הודעה מוכנה.

**הודעת הדירה (מדויק):**
```
היי, שמי {שם הלקוח}. אני מעוניין בפרטים נוספים ולשוחח על סגירת דירת הנופש {שם הדירה}.
```

**הודעת "פרסום דירה" (מדויק):**
```
היי, מעוניין לפרסם את דירת הנופש שלי באתר קל נופש. אפשר לשמוע פרטים?
```

**מימוש טכני (ב-`lib/whatsapp.ts`):**
- בניית הקישור: `https://wa.me/<מספר בפורמט בינלאומי בלי + ובלי תווים>?text=<הודעה מקודדת ב-encodeURIComponent>`.
- המספר לדירה מגיע מהנכס, המספר ל"פרסום" מההגדרות.
- לאמת שהמספר בפורמט בינלאומי תקין לפני בניית הקישור, ולהציג שגיאה ידידותית אם חסר.

## 8. פונקציונליות - אזור המפרסם (admin)

מאחורי התחברות (Netlify Identity). כל פעולת כתיבה עוברת דרך Function מאומתת, ואסור שתתבצע מהלקוח ישירות מול ה-DB.

- **התחברות:** Netlify Identity. גישה ללא התחברות תקינה מנותבת למסך ההתחברות.
- **לוח בקרה:** סיכום קצר (מספר נכסים מפורסמים, נכסים בטיוטה) וקיצורי דרך ל"הוספת נכס" ול"ניהול הגדרות".
- **ניהול נכסים:** רשימה עם סטטוס ופעולות (עריכה, מחיקה, פרסום/השהיה), וטופס הוספה/עריכה עם כל שדות הנכס מפרק 6. בחירת סוג/עיר/תקופות מתוך הרשימות המנוהלות בלבד, לא הקלדה חופשית. כולל שדות הטלפון והווצאפ של הדירה.
- **העלאת תמונות:** מרובות, נשמרות ב-Netlify Blobs, מוצגות דרך Image CDN. ולידציה לסוג וגודל הקובץ.
- **ניהול רשימות החיפוש:** CRUD לסוגי נכס, לערים (כולל שיוך אופציונלי לאזור), ולתקופות. כל שינוי משתקף מיד בשדות הסינון הציבוריים ובטופס הנכס.
- **הגדרות:** מטבע, גבול מחיר תחתון ועליון, ומספר הווצאפ ל"פרסום דירה".

## 9. כללי תוכן וכתיבה

- **שפה:** עברית בלבד, יישור מימין לשמאל (RTL) בכל המסכים, כולל האזור הניהולי.
- **אסור מקפים ארוכים** (התו מקף ארוך, em dash). להשתמש בפסיק, נקודה, סוגריים, או מקף רגיל בלבד.
- **אסור שפה גנרית של AI.** להימנע מניסוחים שיווקיים ריקים ומליצות. הטון: ישיר, אנושי, מוחשי, קצר. לדוגמה, לא "פתרון מהפכני שישדרג את חווייתכם", אלא משפט שמסביר בדיוק מה הדירה מציעה.
- **ריכוז טקסטים:** כל מחרוזות הטקסט יושבות בקובץ אחד (`src/content/copy.he.ts`), כדי שיהיה קל לערוך אותן בלי לגעת בקוד. כולל הכותרות, ה-CTA, הודעות הווצאפ, ה-placeholders והודעות השגיאה.
- המחרוזות המדויקות (כותרות, CTA, הודעות ווצאפ) מופיעות בפרק 7 ויש להעתיק אותן כלשונן.

## 10. דרישות לא-פונקציונליות

- **RTL ועברית:** בכל מקום.
- **רספונסיביות:** mobile-first, ותמיכה מלאה בטאבלט ובדסקטופ. חוויית המובייל בעדיפות גבוהה.
- **ביצועים:** יצירה סטטית של דפים ציבוריים היכן שאפשר, אופטימיזציית תמונות (Image CDN), טעינה עצלה, מינימום JavaScript בצד הלקוח, ושאילתות יעילות.

## 11. תרגום העקרונות לפרקטיקה

**אמינות:**
- ולידציה כפולה (לקוח + שרת) עם סכמה משותפת.

- בדיקות בסיסיות אוטומטיות: lint, type-check, ובדיקות לוגיקת הסינון ולבניית קישורי הווצאפ.

- שום רכיב UI לא ניגש ישירות לנתונים או לסודות.

**מהירות:**

- תמונות דרך Image CDN, גדלים מותאמים, טעינה עצלה.

## 12. מפת דרכים לבנייה

נגזרת מתוכנית הפיתוח ומותאמת למחסנית הזו. סדר לפי תלויות: התשתית והנתונים קודם, אחר כך הניהול, ורק אז החיפוש הציבורי שתלוי ברשימות.

1. **תשתית:** הקמת הפרויקט, חיבור ל-Netlify, `netlify.toml`, חילוץ design tokens מתבנית ה-HTML, והקמת `copy.he.ts`.
2. **שכבת נתונים:** הקמת ה-Postgres והסכמות, שכבת `lib/data`, ו-Functions בסיסיות לקריאה.
3. **ניהול - התחברות והגדרות:** Netlify Identity, מסך התחברות, ו-CRUD לרשימות החיפוש ולהגדרות.
4. **ניהול - נכסים:** טופס נכס מלא, סטטוס, והעלאת תמונות ל-Blobs.
5. **ציבורי - דף בית ותצוגה:** Header, Hero עם הכותרות, כרטיס נכס, ואזור נכסים מוצגים. כולל ה-CTA "פרסום דירה".
6. **ציבורי - מנוע חיפוש:** טופס החיפוש, סינון מצטבר, דף תוצאות, מצב "אין תוצאות".
7. **ציבורי - דף נכס וזרימת ווצאפ:** דף נכס מלא, מודאל שם, ובניית קישורי ווצאפ.
8. **ליטוש לא-פונקציונלי:** רספונסיביות, ביצועים, נגישות, SEO.
9. **בדיקות (QA):** פונקציונליות, נכונות סינון, תאימות, נגישות, ביצועים, אבטחה.
10. **פריסה והשקה:** העלאה ל-Netlify, בדיקות שפיות, והזנת תוכן ראשוני.

## 13. פריסה ל-Netlify

- `netlify.toml`: הגדרת `build command`, תיקיית הפלט, ותיקיית `netlify/functions`. הגדרת redirects לפי הצורך.
- **משתני סביבה (Netlify env vars):** מחרוזת החיבור ל-Netlify Database וכל מפתח רגיש. לעולם לא בריפו.
- **Netlify Identity:** להפעיל בפרויקט, להגדיר את משתמש המפרסם, ולהגן על נתיבי ה-admin ועל ה-Functions שמבצעות כתיבה.
- **Netlify Blobs:** store ברמת ה-site (site-scoped) לתמונות, כדי שיישמרו בין פריסות.
- **Netlify Database:** הקמה והרצת המיגרציות של הסכמה.
- האתר צריך להיבנות ולהיפרס ישירות מ-Netlify, בלי שלבים ידניים מחוץ לפלטפורמה.

## 15. סיכום החלטות

**ננעל:**
- ארכיטקטורה: אפליקציה מלאה ל-Netlify, עם Functions, Netlify Database (Postgres), Netlify Blobs (תמונות), ו-Netlify Identity (התחברות מפרסם). הכול בתוך הפלטפורמה, סודות ב-env vars בלבד.
- שם המותג: קל נופש.
- כותרת ראשית (שתי שורות): `מקום שמרגיש שלכם -` / `מהרגע שנכנסים.`
- כותרת משנית: `לסגור בקלות ובמהירות את דירת הנופש המושלמת שלכם.`
- יצירת קשר לכל דירה: טלפון וּווצאפ בלבד, עם זרימת שם והודעה מוכנה.
- CTA "פרסום דירה" בדף הבית, בולט אך לא חוסם, מוביל לווצאפ עם הודעה מוכנה.
- בלי מקפים ארוכים ובלי שפה גנרית של AI.
- העיצוב מקובץ ה-HTML (שפת עיצוב בלבד), הפונקציונליות מהאפיון.

**ניתן לכוונון בהמשך (לא חוסם):**
- ניסוח מדויק של הודעות הווצאפ (מרוכז ב-`copy.he.ts`).

- מודל "תקופה": כרגע קטגוריות לפי האפיון. ה-DB מתוכנן כך שניתן יהיה להוסיף טווח תאריכים בעתיד.
- מודל ריבוי מפרסמים: מחוץ להיקף גרסה 1, אך ה-DB מאפשר הרחבה אליו.
